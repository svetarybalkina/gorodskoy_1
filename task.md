# Задачи реализации MVP

Документ разбивает `project-requirements.md` на автономные этапы разработки. Пока все задачи из этого файла не реализованы и не приняты владельцем проекта, `AGENTS.md`, `project-history.md`, `project-requirements.md` и `task.md` нужно читать в начале каждой рабочей сессии.

## 1. Базовый каркас проекта и локальный запуск

**Что нужно сделать:** Создать структуру проекта FastAPI по ТЗ: `app/`, `tests/`, `alembic/`, папки `data/`, `imports/`, `exports/`, `logs`; добавить `Dockerfile`, `docker-compose.yml`, `.env.example`, `.gitignore`, базовую конфигурацию, логирование и флаг `ADS_ENABLED=false`. Настроить запуск сайта и админки через `docker compose up`; Telegram-бот должен отключаться с понятным логом, если `TELEGRAM_BOT_TOKEN` не задан.

**Что будет результатом:** Проект запускается одной командой, открывается базовое FastAPI-приложение, приватные файлы исключены из Git, есть документированная команда запуска и минимальный smoke-тест.

## 2. Безопасный перенос проекта в путь с латиницей

**Что нужно сделать:** Аккуратно перенести рабочий проект из пути с кириллицей в путь с латиницей, например `C:\Projects\gorodskoy`, не меняя русские тексты интерфейса, ТЗ и пользовательских документов. Переименовать только технические имена, которые могут влиять на инструменты: файл `Бриф.txt` в `brief.txt`, при необходимости привести `Agents.md` к стандартному `AGENTS.md`, обновить внутренние ссылки в документации и проверить Docker Compose, pytest и локальный запуск уже из нового пути. Старую папку не удалять до успешной проверки новой.

**Что будет результатом:** Проект работает из ASCII-пути без зависимости от кириллических имен в файловой системе; `docker compose up`, `docker compose run --rm app pytest` и локальный smoke-запуск проходят из новой папки; документация указывает актуальные пути и имена файлов.

## 3. База данных, миграции и слой доступа к данным

**Что нужно сделать:** Описать модели и миграции для основных сущностей: материалы, источники, темы, категории, обезличенные вопросы, варианты формулировок, связи вопрос-материал, партии импорта, отчеты импорта, события обезличивания, проверки ФИО, проблемные запросы, внутренние заметки и настройки. Вынести доступ к данным в отдельный слой, чтобы бизнес-логика не зависела напрямую от SQLite и можно было позже перейти на PostgreSQL.

**Что будет результатом:** После применения миграций создается рабочая SQLite-база с минимальной схемой MVP; тесты проверяют создание ключевых сущностей, статусы и общие ограничения данных.

## 4. Публичный сайт и юридические страницы

**Что нужно сделать:** Реализовать главную страницу с поиском, единым блоком категорий ЖКХ, одним пунктом "Животные" внутри этого блока перед категорией "Другое", блоком частых тем, коротким объяснением сервиса и юридическим дисклеймером. Добавить страницу результата с датой, источником, типом материала, официальным текстом, похожими материалами, ссылкой на источник при наличии и кнопкой "Ответ не подошел". Добавить рабочие юридические страницы MVP с явными заглушками неизвестных реквизитов.

**Что будет результатом:** Пользователь без регистрации видит сайт, может открыть главную, перейти по категориям, увидеть страницу результата и юридические документы; интерфейс не называет сервис официальным ресурсом администрации.

## 5. Админка, авторизация и управление карточками

**Что нужно сделать:** Реализовать вход одного администратора по логину и паролю из окружения, защищенную cookie-сессию, список карточек, карточку материала, фильтры по статусам и категориям, смену статусов, скрытие, архивирование, изменение категории и внутренние заметки. Публичной регистрации, ролей и суперадмина не делать.

**Что будет результатом:** Админка доступна только после входа; администратор может управлять материалами и заметками, а скрытые, архивные, дубль-карточки и карточки на проверке не показываются публично.

## 6. Импорт Telegram JSON и отчет обработки

**Что нужно сделать:** Реализовать загрузку стандартной JSON-выгрузки Telegram через админку, разбор сообщений, определение официальных источников, выделение официальных ответов и постов, сохранение исходных данных и ошибок обработки. Сформировать отчет импорта со статистикой, ошибками, найденными материалами, спорными случаями и возможностью скачать отчет. HTML-выгрузку Telegram не поддерживать.

**Что будет результатом:** Администратор загружает JSON, видит итоговый отчет обработки, может скачать его; ошибки отдельных записей не ломают весь импорт.

## 7. Обезличивание и ручная проверка персональных данных

**Что нужно сделать:** Сделать автоматическое обезличивание очевидных персональных данных: телефонов, email, квартир, адресов, номеров обращений и других признаков из ТЗ. Официальные телефоны, адреса, email, сайты и названия ведомств/организаций, нужные для обращения граждан, не скрывать. Оригинальный официальный текст хранить отдельно от публичной версии. Спорные случаи и ФИО сотрудников отправлять на ручную проверку администратору.

**Что будет результатом:** Персональные данные закрываются до дальнейшей обработки, публичная версия не раскрывает очевидные персональные данные, официальный текст сохраняется без изменений, администратор видит очередь спорных случаев и ФИО.

## 8. Локальный поиск, словари и проблемные запросы

**Что нужно сделать:** Реализовать общий поисковый сервис на SQLite FTS с нормализацией текста, локальной морфологической нормализацией русского языка, базовыми словарями ЖКХ, включая категорию "Животные", фильтрацией по категориям и уровнями уверенности совпадения. Добавить запись проблемных запросов через кнопку "Ответ не подошел" и при неуверенной выдаче. Реализовать кандидаты в новые маркеры и варианты формулировок, которые не влияют на публичный поиск без подтверждения администратора. Для админской модерации предусмотреть группировку одинаковых и близких кандидатных формулировок, чтобы подтверждение шло пакетами, а не по одной фразе.

**Что будет результатом:** Сайт находит опубликованные материалы ЖКХ, включая категорию "Животные", не показывает материалы неактивных тем, сохраняет проблемные запросы и предлагает администратору сгруппированные кандидаты для словаря.

## 9. Защищенное удаление и финальные административные сценарии

**Что нужно сделать:** Добавить полный сценарий удаления карточки с защитой от случайного действия, проверками связей и понятным подтверждением. Доработать ручную проверку ФИО, управление категориями, подтверждение новых категорий и маркеров, повторную обработку материалов после исправлений.

**Что будет результатом:** Администратор может безопасно удалить карточку, завершить проверки, подтвердить или отклонить новые категории и маркеры; изменения сразу отражаются на сайте, а после реализации Telegram-бота будут использоваться и ботом.

## 10. Базовая приемка браузерного приложения, тесты и документация запуска

**Что нужно сделать:** Заменить заглушки в юридических документах на финальные значения перед публичным запуском браузерного приложения либо явно зафиксировать, что публичный запуск невозможен до замены. Проверить `SERVICE_NAME`, `PUBLIC_BASE_URL`, контакты и email. Добавить полный набор тестов браузерной части из ТЗ: импорт, обезличивание, сохранение официального текста, публичную выдачу, исключение комментариев жителей, статусы, поиск, проблемные запросы, авторизацию, секреты и smoke-запуск. Описать запуск и тестирование в README.

**Что будет результатом:** Браузерная часть сервиса проверена на уровне локальной демонстрации, команда `docker compose run --rm app pytest` проходит, сайт и админка готовы к дальнейшей доработке и демонстрации; финальная приемка перед публичным запуском остается отдельной поздней задачей MVP.

## 11. Практические рекомендации из официальных материалов

**Что нужно сделать:** Доработать публичную выдачу так, чтобы сервис отвечал не только списком похожих официальных материалов, но и справочным блоком "Что делать" / "Что указано в найденных материалах". Рекомендации должны извлекаться только из сохраненных официальных текстов и связанных с ними данных, без свободного сочинения новых ответов от имени администрации. Нужно выделять практические действия: куда обратиться, какую службу или организацию упоминает администрация, какие условия, сроки, ограничения или дальнейшие шаги названы в официальном материале. Официальный текст материала по-прежнему хранить и показывать без редактирования, кроме уже предусмотренного скрытия персональных данных; справочный блок должен быть явно отделен от официального текста и юридически аккуратно подписан как вспомогательное извлечение из найденных материалов.

Реализовать это без внешних AI API в рамках локальной логики MVP: правила, словари, шаблоны извлечения действий и проверяемые эвристики. Добавить хранение или вычисляемый слой для извлеченных рекомендаций, команду переобработки уже загруженных материалов, обновление поискового ранжирования с приоритетом материалов, где есть действия по запросу пользователя, и тесты на типовые вопросы: "Что делать, если нет горячей воды?", "Куда обращаться по мусору?", "Куда жаловаться на управляющую компанию?", "Что делать при аварии на сетях?". Если по запросу нельзя надежно извлечь конкретные действия, интерфейс должен честно показывать только найденные официальные материалы и предупреждать о неточном совпадении.

**Что будет результатом:** Пользователь видит вверху выдачи короткий практический блок с действиями, извлеченными из официальных материалов, а ниже - сами официальные тексты с датой, источником и ссылкой. Сервис становится полезнее для вопроса "что делать", но не выдает рекомендации как официальный ответ администрации, не меняет официальный текст и сохраняет юридическую границу справочного сервиса. Все новые сценарии покрыты тестами, а существующая публичная выдача, админка, импорт, обезличивание и поиск продолжают проходить полный `pytest`.

## 12. Системная доработка релевантности поиска на проблемных бытовых запросах

**Что нужно сделать:** Доработать поиск не точечно по отдельным словам вроде "подъезд", а системно по типам ошибок релевантности в живых пользовательских формулировках. Взять реальные проблемные запросы из `problem_queries`, полной Telegram JSON-выгрузки и ручных smoke-проверок, сгруппировать их по классам ошибок: слишком общий бытовой контекст, ложные совпадения по одному слабому слову, недооценка устойчивых фраз, слабая привязка к связанным вопросам жителей, перекос между официальным текстом, рекомендациями и вопросами. На этой основе доработать ранжирование, словари, эвристики совпадения и веса сигналов так, чтобы исправления работали сразу на группах похожих запросов, а не на одном существительном. Нужен регрессионный набор тестов по нескольким темам ЖКХ: подъезд, двор, вода, мусор, управляющая компания, квитанции, животные. Для каждой группы надо зафиксировать, какие формулировки должны поднимать релевантные официальные материалы выше нерелевантных. Если найденного официального материала недостаточно для уверенного ответа, интерфейс должен честно показывать, что выдача похожая или неточная, без искусственного притягивания случайной карточки.

**Что будет результатом:** Поиск устойчивее работает на бытовых пользовательских запросах и не требует ручной доработки под каждое новое слово. Проблема вида "грязно в подъезде" исправляется как частный случай общей логики ранжирования и матчинга. В проекте появляется набор регрессионных тестов и контрольных запросов, который защищает от повторного появления того же класса ошибок в других категориях ЖКХ.

## 13. Telegram-бот на общем поисковом сервисе

**Что нужно сделать:** Реализовать Telegram-бота на aiogram, который использует тот же поисковый сервис и ту же базу, что сайт. Добавить команды старта, ввод вопроса, выбор тем и категорий, выдачу официального материала с датой и источником, похожие материалы и аккуратные сообщения, если точного ответа нет.

**Что будет результатом:** При заданном `TELEGRAM_BOT_TOKEN` бот отвечает пользователю теми же опубликованными активными материалами, что и сайт; без токена приложение продолжает работать без бота.

## 14. Подготовка к полной приемочной загрузке Telegram JSON

**Что нужно сделать:** Перед полной приемочной загрузкой Telegram JSON использовать свежую БД или заранее очистить тестовые импортированные черновики, созданные при проверке частичной выгрузки. Очистка должна быть отдельной запускаемой командой с безопасным dry-run по умолчанию и явным подтверждением выполнения. Команда не должна удалять опубликованные материалы, ручные карточки, категории, настройки, источники и данные, не относящиеся к тестовым импортированным черновикам. Если полная выгрузка уже загружается или еще не завершила импорт, никакие массовые операции по БД параллельно не запускать.

**Что будет результатом:** После докачки полной версии JSON владелец проекта может сначала проверить отчет импорта и рост базы, затем при необходимости запустить подготовительную очистку, увидеть список затрагиваемых записей и только потом выполнить очистку без сброса всей рабочей среды. Полная приемочная загрузка не получает дубли из-за частичного тестового импорта и становится основной базой для оценки качества поиска.

## 15. Пост-импортный техпроход после полной загрузки Telegram JSON

**Что нужно сделать:** После завершения полной загрузки Telegram JSON выполнить один аккуратный технический проход по базе в строгом порядке: проверить карточку и отчет импорта; принять решение, нужна ли очистка тестового частичного импорта; при необходимости выполнить очистку отдельным шагом; восстановить или пересобрать связки `resident_questions` -> `material_links`; пересобрать рекомендации из официальных материалов; при необходимости пересчитать классификацию материалов; перестроить поисковый индекс; зафиксировать метрики `app.db`, `materials`, `resident_questions`, `material_links`, время типового поиска и время пересборки индекса. До завершения импорта не запускать эти массовые операции параллельно с записью в SQLite.

**Что будет результатом:** Полная выгрузка становится новой стабильной рабочей базой, поиск и связанные данные синхронизированы после импорта, а владелец проекта получает фактические метрики, по которым можно оценивать качество поиска и запас SQLite без преждевременного перехода на PostgreSQL.

## 16. Финальная приемка перед публичным деплоем

**Что нужно сделать:** После завершения системной доработки поиска, Telegram-бота и post-import техпрохода провести отдельную финальную приемку всей рабочей версии перед production-деплоем. Проверить сквозные пользовательские и административные сценарии на актуальной полной базе: поиск по типовым вопросам, пограничные случаи неточного совпадения, категории, карточки материалов, блок "Что указано в найденных материалах", запись проблемных запросов, работу Telegram-бота, импорт и повторную обработку, очереди проверок, публикацию, скрытие, архивирование и защищенное удаление. Отдельно перепроверить юридические тексты, заглушки, название сервиса, `PUBLIC_BASE_URL`, контакты, правила обезличивания, логику показа похожих материалов и устойчивость поиска на заранее согласованном списке контрольных запросов. Все найденные дефекты должны либо быть исправлены до деплоя, либо явно зафиксированы как блокеры запуска.

**Что будет результатом:** У владельца проекта есть отдельный финальный приемочный этап между разработкой и production-деплоем. К моменту запуска известно, что сайт, админка, поиск, рекомендации, бот, импорт и юридические страницы проверены именно как целостный продукт на актуальных данных, а не только как набор ранее выполненных задач.

## 17. Публичный запуск на домене и production-деплой

**Что нужно сделать:** Когда владелец проекта решит запускать сервис публично, запросить все недостающие данные и доступы: финальные `SERVICE_NAME` и `PUBLIC_BASE_URL`, владельца/оператора сервиса, контакты оператора, email для обращений по персональным данным и удалению информации, подтверждение юридической готовности текстов, домен, выбранный сервер или хостинг, способ доступа к серверу, параметры DNS, email для TLS-сертификата, production-значения `.env`, учетные данные администратора, источник Telegram JSON и решение по первичной базе. После получения данных заменить все юридические заглушки, убрать блокер публичного запуска из документации, подготовить production-конфигурацию без попадания секретов в Git, выполнить полный прогон тестов, развернуть приложение на выбранном сервере через Docker Compose или согласованный хостинг, применить миграции, настроить HTTPS, проверить домен, публичные страницы, админку, импорт, поиск, логи, резервное копирование базы и процедуру отката. Если доступов или обязательных реквизитов не хватает, остановиться и явно перечислить, что блокирует запуск.

**Что будет результатом:** Сервис опубликован на финальном домене по HTTPS, юридические документы не содержат заглушек, `SERVICE_NAME` и `PUBLIC_BASE_URL` соответствуют публичному адресу, секреты находятся только в серверном окружении, база и приватные файлы не попадают в Git, админка доступна только по логину и паролю, основные пользовательские и административные сценарии проверены на production-адресе, владелец проекта получает команды обслуживания, проверки, резервного копирования и отката.
