# Техническое задание

## Справочный сервис официальных ответов городской администрации

Версия: 1.0
Дата: 2026-06-12
Статус: проектное ТЗ для MVP
Основание: `brief.txt` и уточнение решений по стеку от 2026-06-12

---

## 1. Назначение проекта

Проект представляет собой справочный сервис в формате сайта, Telegram-бота и админки. Сервис помогает пользователям находить уже опубликованные официальные ответы и публикации городской администрации по вопросам ЖКХ и теме животных, включая обращения о безнадзорных животных.

Сервис не является официальным ресурсом администрации, не принимает обращения граждан, не оказывает юридические услуги и не формирует новые ответы от имени администрации.

Главная задача MVP: дать пользователю быстрый поиск по базе официальных материалов ЖКХ и теме животных с сохранением юридически аккуратной границы между официальным текстом и вспомогательным интерфейсом сервиса.

---

## 2. Ключевые принципы

1. Официальный текст не редактируется, не сокращается и не пересказывается.
2. Публично показываются только официальные ответы и официальные посты.
3. Комментарии жителей не показываются в публичных карточках.
4. Вопросы жителей используются только в обезличенном виде как поисковый слой.
5. Персональные данные обезличиваются самым первым этапом обработки.
6. ФИО сотрудников в официальных ответах отправляются на ручную проверку администратору.
7. Дата и источник официального материала обязательны.
8. Ссылка на источник необязательна; если она есть, показывается вторично и не отвлекает от ответа.
9. Сайт, Telegram-бот и админка работают с одной общей базой данных.
10. В MVP публично активна тематика ЖКХ, включая категории по теме животных.
11. Материалы вне активных тем анализируются и сохраняются, но не публикуются пользователям.
12. В MVP реклама выключена, но архитектурно заложена на будущее.

---

## 3. Состав MVP

В MVP входят:

1. Публичный сайт.
2. Telegram-бот.
3. Админка.
4. Общая база данных.
5. Модуль импорта JSON.
6. Модуль обезличивания.
7. Модуль поиска.
8. Отчет обработки JSON.
9. Логи ошибок.
10. Защита секретов и приватных файлов.
11. Скачивание отчета обработки.
12. Хранение обезличенной копии обработанного JSON.

### 3.1. Уточненные границы MVP

Чтобы объем MVP оставался реализуемым одним разработчиком с кодинг-агентом, приняты ограничения:

1. Импорт поддерживает один входной формат: стандартную JSON-выгрузку Telegram. HTML-выгрузка Telegram в MVP не поддерживается.
2. Реклама в MVP не реализуется как пользовательская функция. В проекте нужен только технический флаг `ADS_ENABLED=false`, чтобы позже можно было добавить рекламу без пересмотра архитектуры.
3. В MVP есть только один администратор. Роли, права доступа и суперадмин не реализуются, но код не должен мешать добавить их позже.
4. Юридические документы пишутся внутри проекта как рабочие тексты MVP. Переменные реквизиты, которых пока нет, остаются явными заглушками до финального этапа разработки.
5. Обезличивание работает автоматически для очевидных персональных данных. Спорные случаи отправляются администратору на ручную проверку.
6. Оригинальный официальный текст хранится отдельно от публичной обезличенной версии, чтобы сохранять неизменность источника и при этом не публиковать персональные данные.
7. Официальные контакты для обращения граждан не считаются персональными данными и должны сохраняться в публичной версии, если они относятся к ведомству, организации, управляющей компании, диспетчерской или другому официальному каналу помощи.
8. Отсутствие пользовательского списка категорий, слов-маркеров, домена и финального названия сервиса не блокирует разработку: проект стартует с базовыми значениями по умолчанию, которые уточняются перед публичным запуском.

## 4. Технические решения MVP

### 4.1. Основной стек

Для MVP используется стек:

* backend: Python 3.11+;
* web framework: FastAPI;
* frontend публичного сайта и админки: серверный HTML на Jinja2 с HTMX для точечных интерактивных действий;
* Telegram-бот: aiogram;
* база данных MVP: SQLite;
* ORM / слой данных: SQLAlchemy 2.x;
* миграции: Alembic;
* тесты: pytest;
* запуск локально: Docker Compose.

Причина выбора: стек позволяет держать сайт, админку, импорт JSON, обработку текста и Telegram-бота в одном Python-проекте, упростить локальный запуск и подготовить путь к VPS-деплою.

### 4.2. Запуск одной командой

Приложение должно запускаться локально одной командой:

```bash
docker compose up
```

Команда должна поднимать:

* web-приложение FastAPI;
* публичный сайт;
* админку;
* Telegram-бота, если задан `TELEGRAM_BOT_TOKEN`;
* SQLite-базу в volume или локальной папке, исключенной из Git;
* директории для импортов, экспортов, логов и приватных файлов.

Если `TELEGRAM_BOT_TOKEN` не задан, сайт и админка должны запускаться, а бот должен быть отключен с понятным сообщением в логах.

### 4.3. Поиск

В MVP используется гибридный локальный поиск без внешних API:

* полнотекстовый поиск SQLite FTS;
* нормализация текста;
* локальная морфологическая нормализация русского языка без внешних API, чтобы разные формы слова считались одним смысловым маркером;
* базовые словари категорий, синонимов и вариантов формулировок;
* кандидаты в новые маркеры и варианты формулировок, автоматически найденные в импортированных вопросах, официальных ответах и проблемных поисковых запросах;
* ручное подтверждение или отклонение кандидатов администратором;
* фильтрация по категориям;
* эвристика уверенности совпадения.

Автоматическое добавление новых слов, маркеров, синонимов и категорий в рабочий поиск без подтверждения администратора запрещено.

В публичном интерфейсе запрещено показывать технические термины вроде "векторный поиск", "эмбеддинги", "семантический алгоритм".

### 4.4. Обработка JSON без внешнего ИИ

В MVP импорт и классификация выполняются без внешних AI API:

* регулярные выражения;
* словари признаков категорий;
* правила определения официальных источников;
* правила поиска персональных данных;
* ручная проверка спорных случаев в админке.

Внешний AI API не используется в MVP, чтобы не передавать персональные данные третьим сторонам и не усложнять юридический контур.

### 4.5. Авторизация админки

В MVP один администратор.

Доступ к админке:

* логин и пароль задаются через переменные окружения;
* пароль хранится в виде хеша, если создается локальная учетная запись в базе;
* сессия хранится в защищенной cookie;
* секрет сессии задается через `SECRET_KEY`;
* публичная регистрация отсутствует.

## 5. Роли пользователей

### 5.1. Пользователь

Пользователь работает без регистрации.

Пользователь может:

* задать вопрос на сайте;
* задать вопрос в Telegram-боте;
* выбрать категорию ЖКХ, включая категории по теме животных;
* посмотреть официальный ответ или официальный пост;
* посмотреть похожие материалы;
* нажать "Ответ не подошел";
* переформулировать вопрос.

В MVP нет:

* личного кабинета;
* профиля пользователя;
* истории обращений;
* регистрации.

### 5.2. Администратор

В MVP один администратор без разделения прав доступа.

Администратор может:

* загрузить JSON;
* посмотреть отчет обработки;
* скачать отчет обработки;
* опубликовать готовые карточки;
* оставить материалы черновиками;
* отправить данные на повторную обработку;
* проверять найденные ФИО;
* скрывать материалы;
* архивировать материалы;
* менять категории;
* управлять категориями;
* смотреть проблемные поисковые запросы;
* добавлять внутренние заметки;
* удалять карточку полностью через защищенный сценарий.

### 5.3. Будущие роли и суперадмин

Роли, разделение прав доступа и суперадмин не реализуются в MVP.

В MVP не нужно делать:

* таблицы ролей;
* интерфейс управления администраторами;
* журнал действий администраторов;
* отдельные права на удаление, рекламу или настройки.

Код и структура данных должны быть написаны так, чтобы в будущем можно было добавить роли без полной переделки проекта.

## 6. Официальные материалы

Официальными материалами считаются:

1. Ответы от официального бота администрации.
2. Посты из официального канала администрации.

Название официального бота и данные официального канала задаются в настройках источников.

Для каждого официального материала сохраняются:

* источник: официальный бот / официальный канал;
* дата;
* ID сообщения;
* тип материала: официальный ответ / официальный пост;
* ссылка, если она есть или может быть надежно сформирована;
* оригинальный официальный текст без изменений;
* публичная версия официального текста, где допускается только точечное скрытие подтвержденных персональных данных; официальные телефоны, адреса, email, сайты и названия ведомств для обращения граждан сохраняются;
* категория;
* тема верхнего уровня;
* статус;
* признаки персональных данных;
* признаки ФИО на проверку.

Официальный статус по умолчанию присваивается по источнику. Администратор может вручную снять официальный статус, если система ошиблась.

## 7. Тематики и категории

На этапе импорта система анализирует всю JSON-выгрузку, а не только ЖКХ.

Темы верхнего уровня:

* ЖКХ;
* поликлиники;
* образование;
* социальные вопросы;
* госуслуги;
* транспорт;
* благоустройство;
* животные;
* другие административные темы.

В MVP публично активна тема:

* ЖКХ.

Тема животных в публичном интерфейсе не выводится отдельным блоком. Ее базовые категории показываются внутри блока ЖКХ перед общей категорией "Другое". Техническая тема верхнего уровня "животные" может сохраняться в справочнике как непубличная для совместимости с будущей классификацией и импортами.

Материалы вне активных тем:

* сохраняются в базе;
* получают метку "Будущая тематика" или "Не активна в MVP";
* не публикуются пользователям;
* не участвуют в публичном поиске;
* могут хранить черновые связки вопросов и ответов для будущего расширения.

Базовые категории ЖКХ:

* отопление;
* вода;
* подъезд;
* двор;
* управляющая компания;
* квитанции / начисления;
* безнадзорные собаки;
* отлов безнадзорных животных;
* агрессивные животные;
* приюты и передержка;
* правила содержания животных;
* другое.

Если система находит новые категории, администратор должен подтвердить их добавление. После подтверждения категория появляется на сайте, в Telegram-боте, в базе знаний и в поисковых фильтрах.

## 8. Публичный сайт

### 8.1. Главная страница

Главная страница должна содержать:

* поле поиска с подсказкой "Задайте вопрос по ЖКХ или теме животных";
* единый блок категорий ЖКХ;
* категории по теме животных внутри блока ЖКХ перед категорией "Другое";
* компактный блок популярных тем / часто ищут;
* короткое объяснение сервиса;
* короткий юридический дисклеймер;
* ссылки на юридические документы.

Отдельную страницу "Популярные вопросы" в MVP не делать.

Примеры популярных тем:

* Почему нет отопления?
* Что делать, если нет горячей воды?
* Куда жаловаться на управляющую компанию?
* Куда обращаться по начислениям в квитанции?
* Что делать, если грязно в подъезде?
* Куда обращаться по дворовой территории?
* Куда обращаться по бродячим собакам?
* Что делать, если во дворе агрессивная собака?

### 8.2. Страница результата поиска

Страница результата должна показывать:

* найденный официальный ответ или официальный пост;
* тип материала;
* дату;
* источник;
* официальный текст без изменений;
* предупреждение, если совпадение не точное;
* похожие материалы;
* кнопку "Ответ не подошел";
* ссылку на официальный источник, если доступна.

Комментарии жителей в публичную карточку не включаются.

### 8.3. Источник и ссылка

Дата и источник показываются обязательно.

Ссылка показывается только если:

* она есть в JSON;
* или ее можно надежно сформировать по данным выгрузки;
* она ведет на официальный пост, официальный ответ или официальный комментарий администрации;
* она не ведет на комментарий жителя;
* она не раскрывает персональные данные.

Если ссылки нет, карточка все равно может быть опубликована.

Если ссылка есть, она показывается мелко внизу карточки под спойлером, например:

* "Ссылка на комментарий / пост";
* "Открыть официальный источник".

### 8.4. Юридические ссылки

В публичной части должны быть ссылки на:

* пользовательское соглашение;
* политику конфиденциальности / обработки персональных данных;
* согласие на обработку персональных данных;
* дисклеймер о справочном характере сервиса;
* согласие на cookies / аналитику;
* политику модерации и удаления запрещенной информации.

Правила размещения рекламы в MVP не нужны, пока `ADS_ENABLED=false`.

## 9. Telegram-бот

### 9.1. Старт

При запуске бот отправляет:

"Напишите ваш вопрос по ЖКХ или теме животных либо выберите категорию."

Ниже показываются кнопки:

* Отопление;
* Вода;
* Управляющая компания;
* Квитанции;
* Подъезд;
* Двор;
* Безнадзорные собаки;
* Отлов безнадзорных животных;
* Агрессивные животные;
* Приюты и передержка;
* Правила содержания животных;
* Другое.

### 9.2. Выдача ответа

После вопроса пользователя бот показывает:

* основной официальный ответ / пост;
* дату;
* источник;
* ссылку под спойлером, если доступна;
* 2-3 похожих материала, если они есть;
* кнопку "Ответ не подошел".

Если похожих материалов нет, показывается только основной ответ.

### 9.3. Тон сообщений

Тон Telegram-бота:

* спокойный;
* понятный;
* справочный;
* без канцелярита;
* без шуток;
* без обещаний решить проблему;
* без формулировок, будто бот является официальным ресурсом администрации.

## 10. Поиск и выдача

### 10.1. Индексируемые данные

Поиск идет по:

* официальному тексту;
* обезличенным вопросам жителей;
* дополнительным вариантам формулировок;
* категориям;
* официальным постам.

Публичный поиск MVP работает только по активным материалам публичной темы ЖКХ, включая категории по теме животных внутри нее.

### 10.2. Уровни совпадения

Если совпадение уверенное, интерфейс показывает:

"Найден официальный ответ администрации на похожий запрос."

Если совпадение смысловое, но не точное:

"Материал подобран по смысловому сходству с вашим запросом."

Если найдено по категории:

"Найдено по теме: отопление."

### 10.3. Если точного ответа нет

Если точного ответа нет, система должна:

* показать похожие темы;
* предложить переформулировать вопрос;
* предупредить, что материалы подобраны на основании предположения.

Если вопрос слишком общий, система не должна притягивать первый попавшийся ответ. Нужно задать уточняющий вопрос, показать категории и предложить выбрать тему вручную.

### 10.4. Кнопка "Ответ не подошел"

Кнопка должна быть в каждой карточке результата.

После нажатия пользователь видит:

"Поняли. Попробуйте переформулировать вопрос, выберите категорию вручную или посмотрите похожие материалы."

Кнопки:

* Переформулировать вопрос;
* Выбрать категорию;
* Посмотреть похожие материалы.

Событие сохраняется как проблемный поисковый запрос.

## 11. Проблемные поисковые запросы

После нажатия "Ответ не подошел" система сохраняет:

* обезличенный вопрос пользователя;
* какой официальный ответ был показан;
* категория, если была;
* какие похожие материалы были доступны;
* что пользователь нажал после "Ответ не подошел";
* дата;
* канал: сайт / Telegram-бот;
* уровень совпадения, если используется.

В админке администратор видит:

* обезличенный вопрос;
* выданный ответ;
* причину выбора;
* похожие материалы;
* последующее действие пользователя;
* дату;
* канал.

Действия администратора:

* добавить вариант вопроса к правильной карточке;
* поменять категорию;
* пометить выданный ответ как неподходящий;
* связать запрос с другой карточкой;
* оставить без изменений.

Система не должна полностью самообучаться без контроля. Улучшение поиска происходит по модели: система собирает сигналы, администратор принимает решение, система применяет исправления.

---

## 12. Импорт JSON

### 12.1. Общий процесс

JSON загружается через админку.

В MVP поддерживается только стандартная JSON-выгрузка Telegram. HTML-выгрузка Telegram не поддерживается, потому что ее сложнее надежно парсить, проверять и тестировать.

Если в будущем появятся другие форматы, они должны добавляться отдельными импортерами.

Первым этапом выполняется автоматическое выявление и обезличивание персональных данных.

Только после обезличивания система выполняет:

* распознавание официальных ответов;
* выделение официальных постов;
* выделение вопросов жителей;
* построение связок "вопрос -> официальный ответ";
* группировку похожих связок;
* определение категорий;
* поиск дублей;
* подготовку отчета обработки.

До подтверждения администратора данные не должны автоматически попадать в активную публичную базу.

### 12.2. Отчет обработки

После загрузки и первичной обработки JSON администратор видит отчет:

* всего записей в файле;
* найдено официальных ответов;
* найдено официальных постов;
* найдено вопросов жителей;
* сколько персональных данных скрыто;
* сколько ФИО сотрудников отправлено на проверку;
* сколько дублей найдено;
* сколько карточек готово к публикации;
* сколько карточек требует проверки;
* список ошибок обработки;
* ссылка на скачивание отчета.

После отчета администратор может:

1. Опубликовать готовые карточки.
2. Оставить все черновиком.
3. Отправить все на повторную обработку.

Спорные материалы, ФИО сотрудников, дубли и ошибки не публикуются автоматически.

### 12.3. Хранение JSON

Сырой JSON:

* не хранится в Git;
* не должен постоянно лежать в рабочей папке приложения;
* после обработки должен удаляться из рабочей папки приложения или переноситься в закрытое хранилище вне проекта;
* может храниться отдельно в закрытой папке вне проекта или в защищенном архиве, если это нужно владельцу сервиса.

После обработки в приложении остаются:

* база данных;
* обезличенная копия обработанного JSON;
* отчет обработки.

## 13. Персональные данные

Система должна автоматически скрывать очевидные персональные данные:

* ФИО жителей;
* телефоны;
* email;
* номера квартир;
* точные личные адреса;
* номера обращений, если по ним можно идентифицировать человека;
* ссылки на личные профили;
* другие данные, позволяющие идентифицировать физическое лицо.

### 13.1. Что не нужно скрывать

Не нужно скрывать официальные контактные данные, по которым житель может обратиться за помощью:

* телефоны администраций, ведомств, диспетчерских, аварийных служб, управляющих компаний и ресурсоснабжающих организаций;
* адреса администраций, ведомств, приемных, МФЦ, управляющих компаний, аварийных служб и других официальных организаций;
* официальные email, сайты, названия ведомств, организаций, отделов, служб и должностей;
* номера телефонов горячих линий и единых справочных служб;
* реквизиты официальных каналов связи, если они указаны в официальном ответе или посте.

Такие данные должны оставаться в публичной версии материала, потому что они помогают горожанину понять, куда обращаться.

Если система не уверена, является ли телефон, адрес или email личным контактом человека или официальным контактом организации, карточка получает статус "Требует проверки" и не публикуется автоматически.

Спорные случаи, которые нельзя надежно определить автоматически, не публикуются сразу и отправляются администратору на проверку.

В публичную часть не должны попадать:

* персональные данные жителей;
* комментарии жителей;
* личные контакты людей;
* ссылки на комментарии жителей с персональными данными.

Если пользователь вводит персональные данные, система должна предупредить:

"Пожалуйста, не указывайте ФИО, телефон, номер квартиры и другие персональные данные. Если вы уже ввели такие данные, они не будут сохранены. Для поиска ответа будет использована обезличенная версия вопроса."

---

## 14. ФИО сотрудников

Если система находит ФИО конкретного человека в официальном ответе, карточка уходит администратору на проверку.

Администратор выбирает одно из действий:

1. "Это ФИО официального сотрудника - публикуем": ФИО остается в тексте.
2. "Это персональные данные - скрыть ФИО": ФИО заменяется на `[ФИО скрыто]`.
3. "Скрыть всю карточку": материал не публикуется.

Если скрыто только ФИО, карточка может остаться активной, если остальной текст безопасен.

Точечное скрытие ФИО должно сохранять остальной официальный текст без изменений.

---

## 15. Админка

### 15.1. Разделы

В админке должны быть разделы:

* загрузка JSON;
* отчет обработки;
* список карточек;
* карточки на проверке;
* проверка ФИО сотрудников;
* категории;
* проблемные поисковые запросы;
* настройки источников официальности;
* скрытые / архивные материалы.

### 15.2. Карточка материала

Карточка в админке показывает:

* тип материала: ответ / пост;
* источник: бот / канал;
* дату;
* официальный текст без изменений;
* категорию;
* статус;
* найденные персональные данные, которые были скрыты;
* ФИО сотрудников на проверку;
* связанные вопросы жителей только обезличенно;
* похожие материалы / возможные дубли;
* внутреннюю заметку администратора;
* кнопки действий.

Кнопки действий:

* опубликовать;
* скрыть;
* архивировать;
* изменить категорию;
* подтвердить ФИО сотрудника;
* скрыть ФИО;
* удалить карточку полностью.

Администратор не может свободно редактировать официальный текст.

### 15.3. Внутренние заметки

Администратор может добавлять служебную заметку к карточке.

Пользователи ее не видят.

Примеры:

* "Проверить актуальность после отопительного сезона";
* "Есть похожий ответ от 2025 года";
* "ФИО подтверждено как официальный сотрудник".

## 16. Статусы карточек

Нужны статусы:

1. Черновик: загружена, но не опубликована.
2. Активна: показывается пользователям.
3. Требует проверки: есть спорный момент.
4. Архивная: хранится, но не участвует в основной выдаче.
5. Скрыта: вручную убрана из публичного показа.
6. Дубль: не показывается как отдельная основная карточка.
7. Помечена на удаление: ожидает окончательного удаления.
8. Удалена окончательно: физически удалена из рабочей базы.

---

## 17. Удаление карточек

В MVP администратор может удалить карточку полностью, но действие должно быть защищено.

Процесс:

1. Администратор нажимает "Удалить карточку полностью".
2. Система показывает предупреждение и требует подтверждение.
3. Карточка получает статус "Помечена на удаление".
4. Для окончательного удаления требуется отдельное действие "Удалить окончательно".

В будущем после появления суперадмина право окончательного удаления должно перейти только суперадмину.

---

## 18. Реклама

В MVP реклама выключена и не реализуется как пользовательская функция.

В MVP нужно сделать только технический флаг:

```env
ADS_ENABLED=false
```

Пока флаг равен `false`:

* рекламные блоки не показываются;
* админка рекламы не создается;
* рекламные кампании, сроки размещения, форматы и статусы не реализуются;
* раздел "Правила размещения рекламы" не показывается в публичном меню.

Код не должен смешивать рекламу с официальными ответами. Если реклама будет добавлена в будущем, она должна показываться отдельно и с явной пометкой "Реклама".

## 19. Юридические документы

В проекте должны быть предусмотрены:

1. Пользовательское соглашение.
2. Политика конфиденциальности / обработки персональных данных.
3. Согласие на обработку персональных данных.
4. Дисклеймер о справочном характере сервиса.
5. Согласие на cookies / аналитику.
6. Политика модерации и удаления запрещенной информации.
7. Правила размещения рекламы не входят в MVP, пока `ADS_ENABLED=false`.

В юридических документах нужно указать, что сервис:

* является информационно-поисковым сервисом справочного характера;
* помогает найти официальные ответы и публикации администрации;
* не является официальным ресурсом администрации;
* не принимает обращения граждан;
* не оказывает юридические услуги;
* не гарантирует, что найденный материал полностью подходит к конкретной жизненной ситуации пользователя;
* показывает официальные материалы без изменения текста.

Юридические документы в MVP пишутся как полноценные рабочие тексты внутри проекта. Тексты должны быть простыми, понятными пользователю и согласованными с фактической работой сервиса.

Если на момент разработки неизвестны название сервиса, владелец/оператор, домен, контакты или email для обращений по персональным данным и удалению информации, в юридических документах используются явные заглушки:

* `[НАЗВАНИЕ СЕРВИСА]`;
* `[ВЛАДЕЛЕЦ / ОПЕРАТОР СЕРВИСА]`;
* `[ДОМЕН СЕРВИСА]`;
* `[EMAIL ДЛЯ ОБРАЩЕНИЙ]`;
* `[КОНТАКТЫ ОПЕРАТОРА]`.

На финальном этапе разработки MVP перед публичным запуском нужно обязательно напомнить владельцу проекта заменить эти заглушки на реальные данные.

При публичном коммерческом запуске рекомендуется дополнительно проверить эти документы у профильного юриста.

### 19.1. Правовая база для юридических документов

При подготовке юридических документов MVP нужно учитывать:

1. Конституцию Российской Федерации, статья 29: право свободно искать, получать, передавать, производить и распространять информацию законным способом.
2. Федеральный закон от 27.07.2006 N 149-ФЗ "Об информации, информационных технологиях и о защите информации":
* статья 1: закон регулирует отношения при осуществлении права на поиск, получение, передачу, производство и распространение информации;
* статья 7: общедоступная информация может использоваться при соблюдении установленных законом ограничений;
* статья 8: граждане и организации вправе искать и получать информацию из любых источников при соблюдении закона;
* статья 10: при распространении информации владелец сайта должен указывать достоверные сведения о лице, распространяющем информацию.
3. Федеральный закон от 09.02.2009 N 8-ФЗ "Об обеспечении доступа к информации о деятельности государственных органов и органов местного самоуправления": регулирует доступ к информации о деятельности органов власти и местного самоуправления.
4. Федеральный закон от 02.05.2006 N 59-ФЗ "О порядке рассмотрения обращений граждан Российской Федерации": обращения граждан рассматриваются государственными органами, органами местного самоуправления и уполномоченными организациями; сервис не должен создавать впечатление, что принимает обращения в этом порядке.
5. Федеральный закон от 27.07.2006 N 152-ФЗ "О персональных данных":
* статья 3: определяет персональные данные, обработку и обезличивание;
* статья 5: обработка персональных данных должна быть законной, целевой и не избыточной;
* статья 6: устанавливает условия обработки персональных данных.

Юридически предпочтительная базовая формулировка:

"Сервис является информационно-поисковым сервисом справочного характера. Сервис помогает найти опубликованные официальные ответы и материалы администрации, но не является официальным ресурсом администрации, не принимает обращения граждан и не оказывает юридические услуги."

Причина выбора: термин "поиск" прямо соответствует фактической функции сервиса и правовой логике поиска/получения информации. Слово "справочный" лучше использовать как характеристику результата и тона сервиса, а не как единственное определение правового статуса.

## 20. Безопасность

### 20.1. Секреты

Все секреты хранятся только в переменных окружения.

Нельзя писать токены, пароли и ключи API прямо в коде.

В проекте должны быть:

* `.gitignore`;
* `.env.example`;
* проверка отсутствия обязательных секретов при запуске;
* понятные ошибки конфигурации.

### 20.2. Что не должно попадать в Git

В Git не должны попадать:

* `.env`;
* токены;
* пароли;
* ключи API;
* JSON-выгрузки;
* сырые данные;
* базы SQLite;
* резервные копии;
* логи;
* файлы с персональными данными.

### 20.3. Пример `.env.example`

```env
TELEGRAM_BOT_TOKEN=your_token_here
DATABASE_URL=sqlite:///data/app.db
ADMIN_USERNAME=admin
ADMIN_PASSWORD=change_me
SECRET_KEY=change_me
SERVICE_NAME=Городской справочник
PUBLIC_BASE_URL=http://localhost:8000
```

## 21. База данных

Сайт, Telegram-бот и админка работают с одной общей базой.

Это означает:

* ответы единые;
* категории единые;
* статусы карточек единые;
* изменения из админки сразу влияют и на сайт, и на бота;
* если карточка скрыта, она не показывается нигде;
* если категория добавлена, она доступна и на сайте, и в боте;
* если материал опубликован, его можно найти в обоих интерфейсах.

Для MVP используется SQLite.

Архитектура должна предусматривать переход на PostgreSQL:

* бизнес-логика не должна зависеть от SQLite-специфики;
* доступ к данным идет через отдельный слой;
* миграции ведутся через Alembic;
* SQL-запросы, завязанные на SQLite FTS, должны быть изолированы в поисковом модуле.

### 21.1. Основные сущности

Минимальные сущности:

* `materials`: официальные материалы;
* `sources`: источники официальности;
* `categories`: категории;
* `topics`: темы верхнего уровня;
* `resident_questions`: обезличенные вопросы жителей;
* `question_variants`: альтернативные формулировки;
* `material_links`: связи вопрос -> материал;
* `import_batches`: загрузки JSON;
* `import_reports`: отчеты обработки;
* `redaction_events`: события обезличивания;
* `person_name_reviews`: ФИО на проверке;
* `problem_queries`: проблемные поисковые запросы;
* `admin_notes`: внутренние заметки;
* `settings`: настройки, включая `ADS_ENABLED=false`.

Отдельные сущности `ad_slots` / `ads` в MVP не создаются.

## 22. Нефункциональные требования

### 22.1. Производительность MVP

Целевые показатели для локального MVP:

* поиск по базе до 10 000 материалов: до 1 секунды на типовой машине;
* открытие главной страницы: до 500 мс без учета холодного старта;
* импорт JSON может выполняться дольше, но должен показывать итоговый отчет и ошибки.

### 22.2. Надежность

Система должна:

* не публиковать спорные материалы автоматически;
* не падать целиком при ошибке обработки одной записи;
* сохранять ошибки в отчете импорта;
* логировать технические ошибки без записи персональных данных в открытые логи.

### 22.3. Расширяемость

Архитектура должна позволять:

* добавить PostgreSQL;
* добавить новые темы;
* добавить роли и суперадмина;
* добавить рекламу;
* заменить поисковый модуль на более сложный без переписывания сайта и бота.

Тон:

* понятный;
* спокойный;
* справочный;
* без канцелярита;
* без шуток;
* без сложных технических терминов;

Допустимые фразы:

* "Мы нашли официальный ответ администрации на похожий запрос."

## 25. Тестирование

После реализации каждой функциональности должны добавляться тесты и запускаться все существующие тесты.

Минимальный набор тестов MVP:

* тесты обезличивания телефонов, email, квартир, адресов, номеров обращений;
* тесты того, что официальные телефоны, адреса, email, сайты и названия ведомств не скрываются как персональные данные;
* тесты сохранения официального текста без изменений;
* тесты того, что комментарии жителей не попадают в публичную выдачу;
* тесты импорта JSON на корректных и ошибочных данных;
* тесты статусов карточек;
* тесты поиска по активным материалам ЖКХ, включая категории по теме животных;
* тесты исключения материалов вне активных тем из публичного поиска;
* тесты кнопки "Ответ не подошел" и записи проблемного запроса;
* тесты авторизации админки;
* тесты конфигурации секретов;
* smoke-тест запуска приложения.

Команда тестов должна быть документирована. Предпочтительный вариант:

```bash
docker compose run --rm app pytest
```

## 26. Критерии приемки MVP

MVP считается готовым, если:

1. Проект запускается локально одной командой `docker compose up`.
2. Публичный сайт открывается и позволяет искать материалы ЖКХ, включая категории по теме животных.
3. Telegram-бот отвечает на вопрос и показывает тот же набор активных материалов, что и сайт.
4. Админка доступна только после входа.
5. JSON загружается через админку.
6. Персональные данные обезличиваются до дальнейшей обработки.
7. После импорта администратор видит отчет.
8. Готовые карточки не публикуются без подтверждения администратора.
9. Официальный текст в публичной карточке совпадает с сохраненным официальным текстом, кроме точечного скрытия подтвержденных персональных данных; официальные контакты организаций для обращения граждан сохраняются.
10. Комментарии жителей не показываются публично.
11. Материалы вне активных тем сохраняются, но не участвуют в публичном поиске.
12. Кнопка "Ответ не подошел" сохраняет проблемный запрос.
13. Скрытые, архивные, дубль-карточки и карточки на проверке не показываются пользователям.
14. `.env`, базы, JSON, логи и приватные файлы исключены из Git.
15. Все тесты проходят.

## 27. Что не входит в MVP

В MVP не входят:

* регистрация пользователей;
* личный кабинет пользователя;
* история обращений пользователя;
* несколько администраторов;
* разделение прав доступа;
* полноценный суперадмин;
* журнал действий админов;
* публичная выдача по темам кроме ЖКХ;
* активная реклама;
* админка рекламы;
* рекламные блоки, кампании, сроки размещения и статусы рекламы;
* PostgreSQL как основная база;
* внешний AI API для обработки данных;
* публикация юридических документов с незамененными заглушками названия сервиса, оператора, домена и контактных данных.

## 28. План реализации MVP

Рекомендуемые этапы:

1. Базовая структура проекта, Docker Compose, конфигурация, `.env.example`, `.gitignore`.
2. Модели данных, миграции, слой доступа к данным.
3. Публичный сайт: главная, категории, результат поиска, юридические страницы с рабочими текстами MVP и явными заглушками для неизвестных реквизитов.
4. Админка: авторизация, список карточек, статусы, заметки.
5. Импорт JSON, обезличивание, отчет обработки.
6. Поиск SQLite FTS, словари, проблемные запросы.
7. Telegram-бот с общим поисковым сервисом.
8. Проверка ФИО сотрудников.
9. Удаление карточек с защитой.
10. Технический флаг `ADS_ENABLED=false` без рекламной админки и рекламных блоков.
11. Финальная проверка перед публичным запуском: заменить заглушки в юридических документах, проверить название сервиса, домен, контакты и email для обращений.
12. Тесты, документация запуска, smoke-проверка.

## 29. Открытые данные, которые нужно предоставить перед разработкой

Перед началом реализации нужны:

* название официального бота администрации;
* идентификатор или имя официального канала администрации;
* пример стандартной JSON-выгрузки Telegram;
* список точно официальных источников;

Эти данные можно уточнить позже и они не блокируют старт разработки:

* пользовательский список известных категорий и слов-маркеров;
* домен и финальное публичное название сервиса;
* данные для юридических документов: название сервиса, владелец или оператор, контакты, домен, email для обращений по персональным данным и удалению информации.

Решение по категориям и словам-маркерам:

* проект стартует с базовым словарем из этого ТЗ;
* для ЖКХ используются базовые категории: отопление, вода, подъезд, двор, управляющая компания, квитанции / начисления, безнадзорные собаки, отлов безнадзорных животных, агрессивные животные, приюты и передержка, правила содержания животных, другое;
* категории по теме животных показываются в блоке ЖКХ перед общей категорией "Другое"; отдельный публичный блок "Животные" не создается;
* система использует локальную морфологическую нормализацию русского языка, чтобы формы слов вроде "собака", "собаки", "собакой" сводились к общей основе;
* после импортов система предлагает администратору кандидаты: частые слова, фразы, синонимы, новые варианты вопросов и проблемные запросы, которые не уверенно привязаны к существующей категории;
* администратор не дополняет словарь "из головы", а подтверждает, отклоняет или привязывает предложенные системой кандидаты к существующей категории;
* администратор может вручную добавить слово или фразу только если они встречаются в официальных материалах, вопросах жителей или явно нужны для исправления проблемного поискового запроса;
* без подтверждения администратора кандидаты не влияют на публичный поиск.

Решение по домену и названию:

* рабочее название для разработки: "Городской справочник";
* финальное название хранится в настройке `SERVICE_NAME` и может быть изменено без переписывания кода;
* домен хранится в настройке `PUBLIC_BASE_URL`;
* для локальной разработки используется `PUBLIC_BASE_URL=http://localhost:8000`;
* перед публичным запуском нужно заменить рабочее название и локальный адрес на финальные значения.

## 30. Решения, принятые при подготовке ТЗ

1. Для MVP выбран Python/FastAPI + Jinja2/HTMX + SQLite + aiogram.
2. Локальный запуск должен выполняться через `docker compose up`.
3. Поиск в MVP реализуется локально: SQLite FTS, словари, синонимы, ручные варианты вопросов.
4. Внешние AI API в MVP не используются.
5. Админка MVP имеет одного администратора с логином и паролем из окружения.
6. Архитектура должна предусматривать переход на PostgreSQL и более сложный поиск в будущем.
7. Импорт MVP поддерживает стандартную JSON-выгрузку Telegram; HTML-выгрузка Telegram не поддерживается.
8. Реклама в MVP ограничена техническим флагом `ADS_ENABLED=false`; рекламные блоки, админка рекламы и рекламные сущности не реализуются.
9. Роли и суперадмин в MVP не реализуются; допускается только будущая расширяемость без отдельной ролевой модели.
10. Юридические документы пишутся в проекте как рабочие тексты MVP; неизвестные реквизиты допускается оставить явными заглушками до финального этапа перед публичным запуском.
11. Автоматическое обезличивание закрывает очевидные персональные данные, спорные случаи идут на ручную проверку администратора.
12. Официальные телефоны, адреса, email, сайты и названия ведомств/организаций, нужные жителю для обращения за помощью, не скрываются при обезличивании.
13. В MVP тема животных публично доступна как набор категорий внутри ЖКХ перед общей категорией "Другое"; базовый сценарий - обращения о безнадзорных и агрессивных животных, особенно о бродячих собаках.
14. Для юридических документов выбрана формулировка "информационно-поисковый сервис справочного характера"; она точнее отражает функцию поиска официально опубликованных материалов, чем формулировка только "справочный инструмент".
15. Отсутствие пользовательского списка категорий, слов-маркеров, домена, финального названия и реквизитов юридических документов не блокирует разработку. Для MVP используются базовые словари из ТЗ, рабочее название "Городской справочник", `PUBLIC_BASE_URL=http://localhost:8000` для локальной разработки и явные юридические заглушки, которые обязательно заменяются на финальном этапе перед публичным запуском.
