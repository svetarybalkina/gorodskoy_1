# Бриф / техническое задание

## Справочный сервис официальных ответов администрации

### Сайт + Telegram-бот + админка

---

# 1. Суть проекта

Проект представляет собой справочный сервис в формате сайта и Telegram-бота, который помогает пользователям находить опубликованные официальные ответы и публикации администрации по вопросам ЖКХ.

Сервис не является официальным ресурсом администрации, не принимает обращения граждан и не оказывает юридические услуги. Его задача — помочь пользователю быстрее найти уже опубликованную официальную информацию.

В MVP сервис работает только с тематикой ЖКХ. При этом архитектура должна предусматривать последующее расширение на другие административные темы: поликлиники, образование, социальные вопросы, госуслуги, транспорт, благоустройство и другие направления.

---

# 2. Основной принцип работы

Пользователь задаёт вопрос обычными словами.

Система ищет подходящий официальный материал в базе.

Пользователь получает официальный ответ или официальный пост администрации.

Официальный текст показывается без изменений.

Система не должна генерировать новый ответ от себя, переписывать официальный ответ, сокращать его или смешивать официальный текст с неофициальными пояснениями.

Комментарии жителей в публичные карточки не включаются.

Вопросы жителей используются только в обезличенном виде как поисковый слой.

---

# 3. Что считается официальной информацией

Официальными материалами считаются:

1. Ответы от официального бота администрации.
   Название официального бота будет передано отдельно.

2. Посты из официального канала администрации, выгрузка которого используется для импорта.

По умолчанию официальный статус присваивается автоматически по источнику.
Если система ошибочно распознала материал как официальный, администратор может вручную снять статус официальности.

Для каждого официального материала нужно сохранять:

* источник: официальный бот / официальный канал;
* дату;
* ID сообщения;
* тип материала: официальный ответ / официальный пост;
* ссылку, если она есть в JSON или может быть надёжно сформирована;
* официальный текст без изменений.

---

# 4. MVP

В MVP входят:

## 4.1. Сайт

* главная страница;
* поиск по вопросу;
* категории ЖКХ;
* блок популярных тем / часто ищут;
* страница результата поиска;
* страница “О сервисе”;
* юридический блок;
* админка.

## 4.2. Telegram-бот

* стартовое сообщение;
* ввод вопроса;
* кнопки категорий;
* выдача официального ответа;
* 2–3 похожих материала, если они есть;
* кнопка “Ответ не подошёл”.

## 4.3. Админка

* загрузка JSON;
* отчёт обработки;
* список карточек;
* карточки на проверке;
* проверка ФИО сотрудников;
* категории;
* проблемные поисковые запросы;
* скрытые / архивные материалы;
* удаление карточки с защитой;
* внутренние заметки админа.

## 4.4. Импорт и обработка JSON

* анализ всей JSON-выгрузки;
* обезличивание персональных данных первым этапом;
* выделение официальных ответов и постов;
* выделение связок “вопрос → ответ”;
* определение категорий;
* формирование отчёта обработки.

## 4.5. Поиск

* поиск по официальным материалам;
* поиск по обезличенным вопросам жителей;
* поиск по категориям;
* поиск по альтернативным формулировкам;
* предупреждение при смысловом, но не точном совпадении.

## 4.6. Безопасность

* `.gitignore`;
* `.env.example`;
* секреты только через переменные окружения;
* JSON, базы, логи и секреты не попадают в GitHub;
* персональные данные обезличиваются до обработки.

## 4.7. База данных

* сайт, Telegram-бот и админка работают с одной общей базой;
* на старте допустима SQLite;
* архитектура должна предусматривать переход на PostgreSQL без переписывания логики.

---

# 5. Что не входит в MVP, но закладывается на будущее

В развитие уходят:

* реклама;
* публичное отображение правил рекламы;
* суперадмин;
* несколько администраторов;
* разделение прав доступа;
* журнал действий админов;
* личный кабинет пользователя;
* регистрация пользователей;
* публичная выдача по темам: поликлиники, образование, соцвопросы, госуслуги, транспорт и другие;
* полноценная админка рекламы;
* технические отчёты для суперадмина;
* PostgreSQL как основной вариант для роста.

---

# 6. Тематики материалов

На этапе импорта система анализирует всю JSON-выгрузку, а не только ЖКХ.

Система должна уметь определять темы:

* ЖКХ;
* поликлиники;
* образование;
* социальные вопросы;
* госуслуги;
* транспорт;
* благоустройство;
* другие административные темы.

В MVP публично активируется только ЖКХ.

Материалы не по ЖКХ:

* сохраняются в базе;
* не публикуются пользователям;
* не участвуют в публичном поиске;
* получают статус или метку: “Будущая тематика” / “Не активна в MVP”.

По ЖКХ связки “вопрос → ответ” и альтернативные формулировки используются сразу.

По другим темам связки и варианты вопросов могут сохраняться как черновой задел, но не включаются в поиск MVP.

---

# 7. Роли пользователей

## 7.1. Пользователь

В MVP пользователь работает без регистрации.

Пользователь может:

* задать вопрос на сайте;
* задать вопрос в Telegram-боте;
* выбрать категорию;
* посмотреть официальный ответ или пост;
* посмотреть похожие материалы;
* нажать кнопку “Ответ не подошёл”;
* переформулировать вопрос.

В MVP нет:

* личного кабинета;
* профиля пользователя;
* истории обращений;
* регистрации.

## 7.2. Администратор

В MVP один администратор без разделения прав доступа.

Администратор может:

* загрузить JSON;
* посмотреть отчёт обработки;
* опубликовать готовые карточки;
* оставить материалы черновиками;
* отправить файл на повторную обработку;
* проверять ФИО сотрудников;
* скрывать материалы;
* архивировать материалы;
* управлять категориями;
* смотреть проблемные поисковые запросы;
* добавлять внутренние заметки к карточкам;
* удалить карточку полностью с защитой.

## 7.3. Будущий суперадмин

В MVP суперадмин не реализуется как отдельный интерфейс, но архитектурно закладывается.

В будущем суперадмин сможет:

* всё, что администратор;
* управлять администраторами;
* управлять правами доступа;
* включать и выключать рекламу;
* управлять глобальными настройками;
* смотреть технические отчёты;
* видеть журнал действий админов;
* выполнять окончательное удаление карточек.

---

# 8. Главная страница сайта

На главной странице должны быть:

## 8.1. Поле поиска

Пример:

“Задайте вопрос по ЖКХ”

## 8.2. Кнопки категорий

Базовые категории:

* Отопление;
* Вода;
* Управляющая компания;
* Квитанции;
* Подъезд;
* Двор;
* Другое.

## 8.3. Блок популярных тем / часто ищут

Отдельную страницу “Популярные вопросы” в MVP не делать.

На главной нужен компактный блок.

Примеры:

* Почему нет отопления?
* Что делать, если нет горячей воды?
* Куда жаловаться на управляющую компанию?
* Куда обращаться по начислениям в квитанции?
* Что делать, если грязно в подъезде?
* Куда обращаться по дворовой территории?

## 8.4. Короткое объяснение сервиса

Пример:

“Сервис помогает найти опубликованные официальные ответы и публикации администрации по вопросам ЖКХ.”

## 8.5. Короткий юридический дисклеймер

Пример:

“Сервис является справочным инструментом и помогает найти официальные ответы и публикации администрации. Сервис не является официальным ресурсом администрации и не оказывает юридические услуги.”

## 8.6. Ссылки на юридические документы

В публичной части должны быть ссылки на:

* Пользовательское соглашение;
* Политику конфиденциальности;
* Согласие на обработку персональных данных;
* Дисклеймер о справочном характере сервиса;
* Cookies / аналитика;
* Политику модерации и удаления запрещённой информации.

## 8.7. Правила размещения рекламы

Правила размещения рекламы технически заложить, но публично не показывать до включения рекламного функционала.

---

# 9. Страница результата поиска

Страница результата должна показывать:

* найденный официальный ответ или официальный пост;
* тип материала: официальный ответ / официальный пост;
* дату официального материала;
* источник: официальный бот / официальный канал;
* официальный текст без изменений;
* предупреждение, если совпадение не точное;
* похожие материалы;
* кнопку “Ответ не подошёл”;
* ссылку на официальный источник, если доступна.

Комментарии жителей в публичную карточку не включаются.

---

# 10. Дата, источник и ссылка

Дата официального ответа или поста показывается обязательно.

Источник показывается обязательно:

* официальный бот;
* официальный канал администрации.

Ссылка на официальный источник является необязательной.

Ссылка показывается только если:

* она есть в JSON;
* или её можно надёжно сформировать по данным выгрузки;
* она ведёт на официальный пост, официальный ответ или официальный комментарий администрации;
* она не ведёт на комментарий жителя;
* она не раскрывает персональные данные.

Если ссылки нет, карточка всё равно публикуется.
Отсутствие ссылки не считается ошибкой.

Если ссылка есть, она показывается мелко внизу карточки под спойлером.

Пример:

“▼ Ссылка на комментарий / пост”
“Открыть официальный источник”

Ссылка не должна визуально отвлекать пользователя от основного ответа.

---

# 11. Категории

Базовые категории ЖКХ:

* отопление;
* вода;
* подъезд;
* двор;
* управляющая компания;
* квитанции / начисления;
* другое.

Если при анализе JSON система находит новые категории, она предлагает админу добавить их.

Примеры новых категорий:

* лифт;
* канализация;
* ТКО / мусор;
* капремонт;
* домофон;
* подвал;
* крыша;
* парковка.

После подтверждения админом новые категории появляются:

* на сайте;
* в Telegram-боте;
* в базе знаний;
* в поисковых фильтрах.

---

# 12. Telegram-бот

## 12.1. Стартовое сообщение

При запуске бот пишет:

“Напишите ваш вопрос по ЖКХ или выберите категорию.”

Ниже кнопки:

* Отопление;
* Вода;
* Управляющая компания;
* Квитанции;
* Подъезд;
* Двор;
* Другое.

## 12.2. Выдача ответа

После вопроса пользователя бот показывает:

* основной официальный ответ / пост;
* дату;
* источник;
* ссылку под спойлером, если доступна;
* 2–3 похожих материала, если они есть;
* кнопку “Ответ не подошёл”.

Если похожих материалов нет, показывается только основной ответ.

## 12.3. Тон сообщений

Тон Telegram-бота должен быть:

* спокойный;
* понятный;
* справочный;
* без канцелярита;
* без шуток;
* без обещаний решить проблему;
* без формулировок, будто бот является официальным ресурсом администрации.

---

# 13. Поиск

Пользователь вводит вопрос обычными словами.

Система ищет подходящий официальный материал, а не генерирует ответ от себя.

Поиск идёт по:

* официальному тексту;
* обезличенным вопросам жителей;
* дополнительным вариантам формулировок;
* категориям;
* официальным постам.

Если совпадение уверенное, пользователь видит:

“Найден официальный ответ по вашему вопросу.”

Если совпадение смысловое, но не точное:

“Материал подобран по смысловому совпадению.”

Если найдена тема:

“Найдено по теме: отопление.”

Нельзя показывать пользователю технические термины вроде:

* векторный поиск;
* эмбеддинги;
* семантический алгоритм;
* модель сходства.

---

# 14. Если точного ответа нет

Если точного ответа нет, система должна:

* показать похожие темы;
* предложить переформулировать вопрос;
* предупредить, что материалы подобраны на основании предположения.

Если вопрос слишком общий, например:

* “проблема с ЖКХ”;
* “куда жаловаться”;
* “УК ничего не делает”;

система не должна притягивать первый попавшийся ответ.

В этом случае нужно:

* задать уточняющий вопрос;
* показать кнопки категорий;
* предложить пользователю выбрать тему вручную.

---

# 15. Кнопка “Ответ не подошёл”

Кнопка “Ответ не подошёл” должна быть в каждой карточке результата.

После нажатия пользователь видит сообщение:

“Поняли. Попробуйте переформулировать вопрос, выберите категорию вручную или посмотрите похожие материалы.”

Кнопки:

* Переформулировать вопрос;
* Выбрать категорию;
* Посмотреть похожие материалы.

---

# 16. Проблемные поисковые запросы

После нажатия “Ответ не подошёл” система сохраняет проблемный случай для админа.

Сохраняется:

* обезличенный вопрос пользователя;
* какой официальный ответ был показан;
* категория, если была;
* какие похожие материалы были доступны;
* что пользователь нажал после “Ответ не подошёл”;
* дата;
* канал: сайт / Telegram-бот;
* уровень совпадения, если используется.

В разделе админки “Проблемные поисковые запросы” админ видит карточку проблемы:

* обезличенный вопрос пользователя;
* какой официальный ответ был выдан;
* почему система его выбрала;
* какие похожие материалы были рядом;
* что пользователь сделал после нажатия;
* дата;
* канал: сайт / Telegram-бот.

Действия админа:

* добавить вариант вопроса к правильной карточке;
* поменять категорию;
* пометить выданный ответ как неподходящий;
* связать запрос с другой карточкой;
* оставить без изменений.

Система не должна самообучаться полностью без контроля.
Используется модель управляемого улучшения:

система собирает сигналы → администратор принимает решение → система применяет исправления.

---

# 17. Импорт JSON

JSON сначала попадает в приложение для обработки.

Первым этапом выполняется автоматическое выявление и обезличивание персональных данных.

Только после обезличивания система начинает:

* распознавать официальные ответы;
* выделять официальные посты;
* выделять вопросы жителей;
* строить связки “вопрос → официальный ответ”;
* группировать похожие связки;
* определять категории;
* находить дубли;
* готовить отчёт обработки.

До подтверждения администратора данные не должны автоматически попадать в активную публичную базу.

---

# 18. Отчёт обработки JSON

После загрузки и первичной обработки JSON администратор видит отчёт.

Отчёт должен показывать:

* всего записей в файле;
* найдено официальных ответов;
* найдено официальных постов;
* найдено вопросов жителей;
* сколько персональных данных скрыто;
* сколько ФИО сотрудников отправлено на проверку;
* сколько дублей найдено;
* сколько карточек готово к публикации;
* сколько карточек требует проверки;
* список ошибок обработки;
* возможность скачать отчёт файлом.

После отчёта администратор может:

1. Опубликовать готовые карточки.
2. Оставить всё черновиком.
3. Отправить всё на повторную обработку.

Спорные материалы, ФИО сотрудников, дубли и ошибки не публикуются автоматически.

---

# 19. Хранение JSON

Сырой JSON не хранится в GitHub.

Сырой JSON не должен постоянно лежать в рабочей папке приложения.

Оригинал может храниться отдельно:

* в закрытой папке вне проекта;
* в защищённом архиве;
* в месте, не подключённом к Git.

В приложение попадает копия для обработки.

После обработки в приложении остаются:

* база данных;
* обезличенная копия обработанного JSON;
* отчёт обработки.

Сырой оригинал можно хранить отдельно вне проекта, чтобы при необходимости не скачивать его заново.

---

# 20. Персональные данные

Персональные данные должны обезличиваться самым первым этапом.

Система должна автоматически скрывать:

* ФИО жителей;
* телефоны;
* email;
* номера квартир;
* точные личные адреса;
* номера обращений, если по ним можно идентифицировать человека;
* ссылки на личные профили;
* другие данные, позволяющие идентифицировать физическое лицо.

В публичную часть не должны попадать:

* персональные данные жителей;
* комментарии жителей;
* личные контакты людей;
* ссылки на комментарии жителей с персональными данными.

Если пользователь сам вводит персональные данные, система должна предупредить:

“Пожалуйста, не указывайте ФИО, телефон, номер квартиры и другие персональные данные. Если вы уже ввели такие данные, они не будут сохранены. Для поиска ответа будет использована обезличенная версия вопроса.”

---

# 21. Комментарии жителей

Комментарии жителей из JSON не включаются в публичные карточки.

Они используются только:

* для понимания формулировок вопросов;
* для поиска;
* для формирования связок “вопрос → официальный ответ”;
* для альтернативных вариантов вопросов.

Принятое правило:

официальный ответ показываем полностью → вопрос жителя используем обезличенно → персональные детали не публикуем и не сохраняем в открытой базе → комментарии жителей в карточки не включаем.

---

# 22. Контактные данные ведомств

В официальных ответах могут содержаться данные ведомств:

* названия;
* адреса;
* телефоны;
* email;
* сайты;
* режим работы.

Такие данные можно публиковать, если они относятся к организации или официальному каналу связи, а не к частному лицу.

Можно показывать:

* название ведомства;
* общий адрес ведомства;
* общий телефон приёмной;
* официальный email;
* официальный сайт;
* режим работы;
* ссылку на официальный источник.

Автоматически обезличивать и не публиковать:

* личный мобильный телефон;
* личную почту;
* данные заявителей;
* номера квартир;
* телефоны жителей;
* точные личные адреса жителей;
* номера обращений, если по ним можно идентифицировать человека.

---

# 23. ФИО сотрудников

ФИО конкретных сотрудников обрабатываются отдельно.

Если система находит ФИО конкретного человека в официальном ответе, карточка уходит админу на проверку.

Админ видит найденное ФИО и выбирает:

1. “Это ФИО официального сотрудника — публикуем”
   ФИО остаётся в тексте.

2. “Это персональные данные — скрыть ФИО”
   ФИО заменяется на “[ФИО скрыто]”.

3. “Скрыть всю карточку”
   Если материал нельзя безопасно показывать.

Если скрыто только ФИО, карточка может остаться активной, если остальной текст безопасен.

Пример:

“Обратитесь к [ФИО скрыто], специалисту отдела…”

Это должно быть реализовано как функция точечного обезличивания ФИО без удаления всей карточки.

---

# 24. Админка

В админке должны быть разделы:

* загрузка JSON;
* отчёт обработки;
* список карточек;
* карточки на проверке;
* проверка ФИО сотрудников;
* категории;
* проблемные поисковые запросы;
* настройки источников официальности;
* скрытые / архивные материалы.

Администратор не может редактировать официальный текст.

Официальный текст либо показывается без изменений, либо скрывается / архивируется.

Администратор может:

* опубликовать карточку;
* скрыть карточку;
* отправить карточку в архив;
* изменить категорию;
* изменить статус;
* подтвердить ФИО официального сотрудника;
* скрыть ФИО;
* добавить внутреннюю заметку;
* удалить карточку полностью с защитой.

---

# 25. Карточка материала в админке

Карточка в админке должна показывать:

* тип материала: ответ / пост;
* источник: бот / канал;
* дату;
* официальный текст без изменений;
* категорию;
* статус;
* найденные персональные данные, которые были скрыты;
* ФИО сотрудников на проверку;
* связанные вопросы жителей — только обезличенно;
* похожие материалы / возможные дубли;
* внутреннюю заметку админа;
* кнопки действий.

Кнопки действий:

* опубликовать;
* скрыть;
* архивировать;
* изменить категорию;
* подтвердить ФИО сотрудника;
* скрыть ФИО;
* удалить карточку полностью.

---

# 26. Внутренние заметки админа

Администратор может добавлять к карточке служебную заметку.

Пользователи её не видят.

Примеры заметок:

* “Проверить актуальность после отопительного сезона”;
* “Есть похожий ответ от 2025 года”;
* “ФИО подтверждено как официальный сотрудник”.

---

# 27. Статусы карточек

Нужны статусы:

1. Черновик
   Загружена, но не опубликована.

2. Активна
   Показывается пользователям.

3. Требует проверки
   Есть спорный момент.

4. Архивная
   Хранится, но не участвует в основной выдаче.

5. Скрыта
   Вручную убрана из публичного показа.

6. Дубль
   Не показывается как отдельная основная карточка.

7. Помечена на удаление
   Ожидает окончательного удаления.

8. Удалена окончательно
   Физически удалена из рабочей базы.

---

# 28. Удаление карточек

В MVP у администратора должна быть функция “Удалить карточку полностью”.

Удаление должно быть защищено от случайного действия.

Нужно предусмотреть:

* подтверждение действия;
* промежуточный статус “Помечена на удаление”;
* отдельное действие “Удалить окончательно”.

В будущем, после появления суперадмина:

* у обычного админа полное удаление закрывается;
* право окончательного удаления остаётся только у суперадмина.

---

# 29. Реклама

В MVP реклама выключена.

Но архитектурно нужно заложить будущий рекламный функционал.

На будущее нужно предусмотреть:

* рекламные блоки;
* включение / выключение рекламы;
* управление рекламой через админку;
* сроки размещения;
* статусы рекламы;
* форматы рекламы;
* правила размещения рекламы;
* контакт для рекламодателей.

Форматы рекламы на будущее:

* текстовый блок;
* баннер + текст;
* карточка рекламодателя;
* рекламная ссылка;
* видеоролик / ссылка на видео / загруженный видеофайл;
* срок размещения;
* статус: черновик / активна / завершена / скрыта.

Пока реклама не активна:

* раздел “Правила размещения рекламы” не показывается в публичном меню;
* страница может существовать технически, но быть скрытой;
* в настройках должен быть переключатель “Реклама включена / выключена”.

Когда реклама включается, появляются:

* рекламные блоки;
* страница “Разместить рекламу”;
* правила размещения рекламы;
* контакт для рекламодателей.

Реклама не должна смешиваться с официальными ответами.

Правильный порядок:

официальный ответ → похожие материалы → рекламный блок с пометкой “Реклама”.

# 30. Юридические документы

В проекте должны быть предусмотрены:

1. Пользовательское соглашение.
2. Политика конфиденциальности / обработки персональных данных.
3. Согласие на обработку персональных данных.
4. Дисклеймер о справочном характере сервиса.
5. Согласие на cookies / аналитику.
6. Политика модерации и удаления запрещённой информации.
7. Правила размещения рекламы — заложить технически, но скрыть до включения рекламы.

В юридических документах нужно указать, что сервис:

* является справочным инструментом;
* помогает найти официальные ответы и публикации администрации;
* не является официальным ресурсом администрации;
* не принимает обращения граждан;
* не оказывает юридические услуги;
* не гарантирует, что найденный материал полностью подходит к конкретной жизненной ситуации пользователя;
* показывает официальные материалы без изменения текста.

---

# 31. Правовая основа для официальных данных ведомств

В юридических документах нужно заложить основания, по которым сервис может показывать справочную информацию об официальных ведомствах.

Учитывать:

* 152-ФЗ “О персональных данных”: персональные данные относятся к физическим лицам; данные ведомств и организаций сами по себе не являются персональными данными.
* 8-ФЗ “Об обеспечении доступа к информации о деятельности государственных органов и органов местного самоуправления”: информация о деятельности госорганов и органов местного самоуправления доступна в установленном законом порядке.
* ГК РФ статья 1259: официальные документы государственных органов и органов местного самоуправления, а также материалы административного характера, не являются объектами авторского права.
* При необходимости учитывать ГК РФ статью 1274 в части цитирования с указанием автора и источника.

Финальные юридические документы должен проверить профильный юрист.

# 32. Безопасность и GitHub

В проекте обязательно должен быть `.gitignore`.

В GitHub не должны попадать:

* `.env`;
* токены;
* пароли;
* ключи API;
* JSON-выгрузки;
* сырые данные;
* базы SQLite;
* резервные копии;
* логи;
* файлы с персональными данными.

В GitHub можно хранить только `.env.example` без настоящих секретов.

Пример `.gitignore`:

.env
.env.*
!.env.example

/imports/
/exports/
/raw_data/
/private/
/backups/

*.db
*.sqlite
*.sqlite3

*.log
**pycache**/
node_modules/

Пример `.env.example`:

TELEGRAM_BOT_TOKEN=your_token_here
DATABASE_URL=sqlite:///app.db
ADMIN_PASSWORD=change_me
SECRET_KEY=change_me

Все секреты должны храниться в переменных окружения.

Нельзя писать токены прямо в коде.

Правильно:

TELEGRAM_BOT_TOKEN = os.getenv("TELEGRAM_BOT_TOKEN")

Неправильно:

TELEGRAM_BOT_TOKEN = "123456:ABC..."

---

# 33. База данных

Сайт, Telegram-бот и админка работают с одной общей базой данных.

Это значит:

* ответы единые;
* категории единые;
* статусы карточек единые;
* изменения из админки сразу влияют и на сайт, и на бота;
* если карточка скрыта, она не показывается нигде;
* если категория добавлена, она доступна и на сайте, и в боте;
* если материал опубликован, его можно найти в обоих интерфейсах.

Для MVP допускается SQLite.

Но архитектуру нужно сделать так, чтобы потом можно было перейти на PostgreSQL.

Требования:

* код не должен быть жёстко привязан к SQLite;
* должен быть отдельный слой работы с базой данных;
* сайт, бот и админка не должны обращаться к базе хаотично напрямую;
* структура данных должна быть совместима с будущим переносом.

# 35. Тон интерфейса

Тон интерфейса:

* понятный;
* спокойный;
* справочный;
* без канцелярита;
* без шуток;
* без сложных технических терминов;
* юридически аккуратный.

Нельзя писать так, будто сервис является официальным ресурсом администрации.

Нельзя обещать пользователю, что сервис решит его проблему.

Примеры подходящих фраз:

“Мы нашли официальный ответ администрации по вашей теме.”

“Материал подобран по смысловому совпадению.”

“Сервис помогает найти опубликованные ответы администрации, но не является официальным ресурсом администрации.”

# 36. Критически важные правила

Эти правила нельзя потерять при разработке:

1. Официальный текст не редактируется.
2. Официальный текст либо показывается без изменений, либо карточка скрывается / архивируется.
3. Комментарии жителей не показываются в публичных карточках.
4. Вопросы жителей используются только обезличенно.
5. Персональные данные обезличиваются самым первым этапом.
6. ФИО сотрудников отправляются админу на проверку.
7. Дата и источник обязательны.
8. Ссылка необязательна.
9. Если ссылка есть, она показывается мелко под спойлером.
10. Если ссылки нет, карточка всё равно публикуется.
11. Сайт и бот работают с одной общей базой.
12. В MVP один администратор.
13. Пользователи работают без регистрации.
14. JSON, `.env`, базы, логи и секреты не попадают в GitHub.
15. Кнопка “Ответ не подошёл” есть в каждой карточке.
16. Реклама заложена на будущее, но скрыта в MVP.
17. Не ЖКХ-темы анализируются и сохраняются, но не публикуются в MVP.
18. Полное удаление карточки в MVP доступно админу с защитой; в будущем только суперадмину.
