# AGENTS.md

## Название проекта
Городской

## Правила работы

1. Перед началом любой сессии читай этот файл целиком, а также все файлы, указанные в разделе "Обязательные для чтения" ниже.
2. После реализации каждой задачи или функционала - пиши тесты и прогоняй ВСЕ предыдущие тесты вместе с новыми. Если что-то сломалось - чини прежде, чем двигаться дальше.
3. Работай аккуратно: ничего не ломай из того, что уже реализовано и протестировано.
4. После каждого крупного блока работы обновляй `project-history.md`: что сделано, где лежит, какие ключевые решения приняты. `AGENTS.md` обновляй только при изменении правил, обязательных документов или процесса работы.
5. До тех пор, пока все задачи из `task.md` не будут реализованы и приняты владельцем проекта, в начале каждой сессии обязательно читай все файлы из раздела "Обязательные файлы для чтения". После полной реализации и приемки всех задач из `task.md` эти файлы можно перестать подгружать как обязательный контекст, если они не нужны для текущей работы.
6. Обязательные русскоязычные Markdown-документы храни в UTF-8 с BOM, чтобы Windows PowerShell 5.1 корректно читал их обычным `Get-Content` без явного `-Encoding UTF8`. При правках `AGENTS.md`, `project-history.md`, `project-requirements.md` и `task.md` не удаляй BOM.
7. После реализации каждого пункта или функционала в итоговом ответе давай владельцу проекта конкретный алгоритм перезапуска приложения и проверки результата: какие команды выполнить, из какой папки, какой URL открыть и на что посмотреть, чтобы увидеть изменение.
8. Если нужно запросить разрешение владельца проекта на действие, сначала в обычном сообщении формулируй прямой вопрос и два полных, понятных варианта ответа для выбора. Во всплывающем системном запросе на разрешение не дублируй длинные варианты, потому что интерфейс может обрезать текст; пиши там только короткий однозначный вопрос о самом действии.

## Чеклист после создания или изменения важных документов

После создания или существенного изменения ТЗ, архитектурного документа, плана задач, схемы данных, правил безопасности или другого опорного документа обязательно проверь:

1. Нужно ли добавить документ в раздел "Обязательные файлы для чтения".
2. Нужно ли обновить историю изменений в `project-history.md`.
3. Не появились ли новые решения, ограничения или команды запуска, которые должны быть явно зафиксированы для следующих сессий.
4. Если документ влияет на разработку или приемку, он должен быть добавлен в обязательные файлы для чтения сразу в том же блоке работы.

## Обязательные файлы для чтения

1. `project-requirements.md` - техническое задание MVP справочного сервиса официальных ответов городской администрации.
2. `task.md` - пронумерованный план автономных этапов реализации MVP по техническому заданию.
3. `project-history.md` - журнал выполненных этапов, проверок, ключевых решений и изменений процесса работы.

## История изменений

Полная история проекта вынесена в обязательный файл `project-history.md`.
