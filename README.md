# Городской

MVP справочного сервиса официальных ответов городской администрации.

## Рабочая папка

Актуальная рабочая копия проекта после задачи 2 находится в ASCII-пути:

```text
C:\Projects\gorodskoy
```

Старая папка `C:\Users\user-pc\Desktop\Нейро\Городской` оставлена как резерв до ручной приемки переноса.

## Локальный запуск

Основная команда запуска по ТЗ:

```bash
docker compose up
```

После запуска приложение доступно на `http://localhost:8000`.

Проверочные URL:

* `http://localhost:8000/`
* `http://localhost:8000/health`
* `http://localhost:8000/admin`

Если `TELEGRAM_BOT_TOKEN` не задан, веб-приложение продолжает запускаться, а Telegram-бот отключается с понятным сообщением в логах.

## Статус публичного запуска

Текущая сборка браузерного приложения готова для локальной демонстрации сайта и админки, но не готова к публичному запуску.

Причина: в юридических документах намеренно оставлены явные заглушки:

* `[НАЗВАНИЕ СЕРВИСА]`
* `[ВЛАДЕЛЕЦ / ОПЕРАТОР СЕРВИСА]`
* `[ДОМЕН СЕРВИСА]`
* `[EMAIL ДЛЯ ОБРАЩЕНИЙ]`
* `[КОНТАКТЫ ОПЕРАТОРА]`

Перед публичным запуском нужно заменить эти значения на реальные данные, проверить `SERVICE_NAME`, `PUBLIC_BASE_URL`, оператора сервиса, контакты оператора и email для обращений по персональным данным и удалению информации. До этой замены публикация юридических документов и публичный коммерческий запуск запрещены.

## Ручная smoke-проверка браузерной части

После сборки и тестов запустите приложение:

```bash
docker compose up -d --build
```

Откройте:

* `http://localhost:8000/` - главная страница с поиском по ЖКХ и категорией "Животные".
* `http://localhost:8000/search?q=отопление` - страница поиска без технических терминов и без непубличных материалов.
* `http://localhost:8000/admin` - редирект на вход в админку, если сессии нет.
* `http://localhost:8000/legal/disclaimer` - юридический дисклеймер с явными заглушками до публичного запуска.

Если проверка завершена, приложение можно остановить:

```bash
docker compose down
```

## Конфигурация

Пример переменных окружения лежит в `.env.example`. Этот файл попадает в GitHub и должен содержать только заглушки, без реальных логинов, паролей, токенов и секретных ключей. Реальные значения храните только в локальном `.env`.

Для локальной разработки по умолчанию используются:

* `SERVICE_NAME=Городской справочник`
* `PUBLIC_BASE_URL=http://localhost:8000`
* `ADS_ENABLED=false`

Для входа в админку задайте в локальном `.env` реальные значения:

```env
ADMIN_USERNAME=...
ADMIN_PASSWORD=...
SECRET_KEY=...
```

После запуска админка доступна по адресу `http://localhost:8000/admin`. Без успешного входа закрытые страницы админки должны перенаправлять на `/admin/login`.

Импорт Telegram JSON в админке `/admin/imports` включается только после настройки официального источника в локальном `.env`:

```env
OFFICIAL_TELEGRAM_SOURCE_ID=official_channel_id_or_username
OFFICIAL_TELEGRAM_SOURCE_NAME=Официальный источник администрации
OFFICIAL_TELEGRAM_SOURCE_KIND=official_channel
```

`OFFICIAL_TELEGRAM_SOURCE_ID` должен совпадать с `id`, `username`, `from_id` или другим идентификатором официального источника из стандартной JSON-выгрузки Telegram. Допустимые значения `OFFICIAL_TELEGRAM_SOURCE_KIND`: `official_channel`, `official_bot`, `telegram_bot`, `website`.

После входа в админку откройте `http://localhost:8000/admin/imports`, загрузите стандартный JSON-экспорт Telegram и проверьте отчет обработки. HTML-выгрузки Telegram в MVP не поддерживаются.

## Массовая публикация готовых импортированных черновиков

После импорта и обезличивания ответы остаются черновиками, пока администратор явно не подтвердит публикацию. Чтобы не публиковать тысячи карточек вручную, откройте:

```text
http://localhost:8000/admin/publication
```

Страница показывает dry-run: сколько импортированных черновиков готово к публикации и сколько карточек заблокировано проверками. Массовая публикация затрагивает только импортированные официальные черновики публичной темы без нерешенных ФИО и без признаков неочищенного персонального обращения. Ручные черновики, карточки на проверке, непубличные темы и небезопасные тексты не публикуются.

Для выполнения действия нужно ввести точную фразу подтверждения, указанную на странице. После публикации поисковый индекс перестраивается, и опубликованные карточки становятся видны на сайте.

## Практические извлечения из официальных материалов

Публичный поиск показывает справочный блок "Что указано в найденных материалах", если из опубликованных официальных текстов удалось надежно извлечь практическое действие: куда обратиться, какую службу упоминает администрация, какие условия, сроки, ограничения или дальнейшие шаги названы в материале.

Извлечения строятся локальными правилами без внешних AI API и отображаются только по `public_text`, то есть по уже обезличенной публичной версии официального материала. Официальный текст карточки не редактируется и остается ниже отдельным блоком.

Для уже загруженной базы можно выполнить dry-run пересборки:

```bash
docker compose run --rm app python scripts/rebuild_recommendations.py
```

Чтобы применить пересборку и обновить поисковый индекс:

```bash
docker compose run --rm app python scripts/rebuild_recommendations.py --execute
```

Дополнительные фильтры:

```bash
docker compose run --rm app python scripts/rebuild_recommendations.py --public-only
docker compose run --rm app python scripts/rebuild_recommendations.py --status active --execute
```

## Связки вопросов жителей с официальными ответами

Поиск использует обезличенные вопросы жителей только как внутренний поисковый слой. Эти вопросы не выводятся в публичных карточках, но помогают понять реальную потребность пользователя и подобрать официальный ответ по смыслу вопроса, а не только по совпадению отдельных слов.

При новом импорте Telegram JSON связки создаются автоматически по `reply_to_message_id`: если официальный ответ администрации был ответом на сообщение жителя, сервис сохраняет обезличенный вопрос в `resident_questions` и связь в `material_links`.

Для уже загруженной базы можно восстановить связки из сохраненного исходного JSON. Сначала выполните dry-run:

```bash
docker compose run --rm app python scripts/rebuild_question_links.py --batch-id 2
```

Чтобы применить восстановление и перестроить поисковый индекс:

```bash
docker compose run --rm app python scripts/rebuild_question_links.py --batch-id 2 --execute
```

Если нужно указать файл явно:

```bash
docker compose run --rm app python scripts/rebuild_question_links.py --batch-id 2 --export-path imports/f45cc6e1426d4ae5af806fb29a654d6d_result.json --execute
```

Файлы `.env`, базы данных, JSON-выгрузки, экспорты и логи исключены из Git.

Реальные выгрузки Telegram с персональными данными нельзя хранить в репозитории. Для ручной загрузки используйте папку вне проекта, например `C:\Projects\gorodskoy_private\telegram_exports`, или временно кладите файл в `imports/`, который исключен из Git и Docker build context.

## Очистка тестового частичного импорта перед полной загрузкой

Перед полной приемочной загрузкой Telegram JSON можно оставить базу как есть, если дубли из частичного теста не мешают отчету. Если нужна чистая приемочная загрузка без дублей от тестовой выгрузки, сначала выполните dry-run:

```bash
docker compose run --rm app python scripts/cleanup_test_imports.py
```

Команда по умолчанию ничего не удаляет. Она показывает, сколько импортированных черновиков, карточек на проверке и дублей будет затронуто для `OFFICIAL_TELEGRAM_SOURCE_ID` из локального `.env`.

Чтобы применить очистку:

```bash
docker compose run --rm app python scripts/cleanup_test_imports.py --execute
```

Очистка удаляет только материалы, созданные импортом и оставшиеся в статусах `draft`, `needs_review` или `duplicate`, а также их служебные зависимые записи. Опубликованные материалы, ручные карточки, источники, категории, настройки, отчеты импорта и приватные JSON-файлы не удаляются.

## Повторное обезличивание существующих карточек

После изменения правил обезличивания можно безопасно проверить, сколько существующих карточек будет обновлено:

```bash
docker compose run --rm app python scripts/reanonymize_materials.py
```

Команда по умолчанию работает в режиме dry-run и ничего не меняет.

Чтобы применить новые правила ко всем существующим материалам:

```bash
docker compose run --rm app python scripts/reanonymize_materials.py --execute
```

Команда пересоздает события обезличивания, обновляет `public_text` из неизменного `original_text` и перестраивает поисковый индекс. Если опубликованная карточка после переобработки требует ручной проверки, она переводится в статус "На проверке".
